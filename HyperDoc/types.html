Types

Types provide the means to specify constraints on a program.

There are two forms of types: enumeration and refinement.

Each type is equivalent to itself.

No two types can be equivalent.

A unit is an instance of an enumeration if it is equal to a member.

A unit is an instance of a refinement if applying the predicate with the unit as a parameter results in a Success judgement.

Enumeration Constructor

The Enumeration Constructor is a default expression that will create a type which is an enumeration.

An Enumeration Constructor will resolve each field of the expression following the keyword type. Each result of a resolution is taken as a member of the enumeration.

Refinement Constructor

The Refinement Constructor is a default expression that creates a type which is a refinement.

Type Arithmetic

Combine

Combine is a default expression which will create a type using the members and predicates of the two parameters.

Combine will create a type which is a supertype of the two parameter types such that ((type a b) + (k | (n -> f n))) results in (type a b (k | (n -> f n))).

Remove

Remove is a default expression that will create a type with all the members of the first parameter and no members which are instances of the second parameter.

The initial type parameter must be an enumeration otherwise it is considered an error.

Derive from Instance

The default expression Derive from Instance will result in the type which constrains a value or an identifier.

The type recognized as the constraint must be relative to the scope where the Derive from Instance expression is evaluated.

Constraints

A constraint implies that there are certain structural properties which a program must exhibit.

Given a program and a constraint, that program may either satisfy the constraint, leading to the program being evaluated, or violate it, leading to an error.

Assertion

A constraint is established via the default expression, Assertion.

Assertion will accept an identifier or program and a type. It will establish a constraint such that the value bound to the identifier or the result of evaluating the program must be an instance of that type.

An assertion on a symbol which is not already an identifier will instantiate that symbol as an identifier and then constrain the resulting identifier.

The result of an assertion will always be the program or identifier which is constrained.

The constraint established by an assertion is relative to the context in which the constrained entity is instantiated. Additionally, a constraint does not apply outside of the context where the assertion was evaluated.

Correctness

A program may only be evaluated if it is correct.

A program is correct if it satisfies all constraints. A conforming implementation must prove that a program satisfies all constraints. The method of proof is implementation dependent.

Abstract Evaluation

A case may arise where an assertion has established an identifier in a scope, and that identifier is then used in an expression before a definite value has been given for that binding. In this case, evaluation of that expression will attempt to produce a type as the result of the expression where each member of that type could be a result if a definite value is substituted for that identifier.

If the type resulting from an abstract evaluation has only one member, then that member can be given directly as a result value.
