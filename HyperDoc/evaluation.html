Evaluation

Evaluation determines the result of a program through a process of repeated intermediary steps which approach a final data result.

Given a program, there are two steps which are performed to determine the result, Rewriting and Reduction. Due to the nature of how a Program Model represents a program, programs will always have one "top" expression which further expressions are nested in. As such, Rewriting and Reduction are defined as they behave when operating on an expression

Rewriting is executed repeatedly to an expression until it is possible to execute Reduction, unless an error occurs.

Rewriting

An expression may either correspond to a default expression, or an expression which can be transformed.

An expression can be transformed if it is a subtype of at least one identifier in a binding which is available in the scope.

If an expression is not a default expression, then the rewrite step will be performed.

If a rewrite step is performed, it will check the image for a transform which can be applied to the expression.

If no transforms can be applied to the expression, and it is not a default expression, it is considered an error.

If an expression with one or more nested expressions is rewritten, each inner expression must be rewritten before the outer expression.

Rewriting must not resolve a binding if that binding is not a transform.

Transforms

A transform is a subtype of function,
(Collection -> *)
such that a transform binds one or more expressions as identifiers, according to a Program Model structure specified by the parameter type of the transform.

The Collection used as the identifier for a transform corresponds directly to Program Model structure of a rewriteable expression.

The Collection used as the identifier for a transform may contain fields which are the types Program, Expression, and Symbol which correspond to specific formations of Program Model components at the syntactic level.
The type, Program as used in a transform corresponds to a program which may result in anything, or even nothing, (as _).
The type, Expression as used in a transform corresponds to a Program Model structure and not an instance of data (Refer to the section on Syntax).
The type, Symbol as used in a transform corresponds to a symbol as a Program Model component (Refer to the section on Syntax).

In order to resolve a binding that is a transform, it must be applied to the Program Model component which was recognized as an identifier by the rewriting step. The Program Model component which was used as the parameter will be replaced in-place in the program with the result of the transform application.

The result of resolving a transform must be an instance or subtype of Program, otherwise it must be treated as data during reduction.

Reduction

A program can be reduced only when all expressions in the program are default expressions.

A program consisting of only default expressions, it may only be reduced if it is determined to be correct.

Reduction begins by attempting to resolve the identifiers in the expression being reduced.

Reduction will substitute all identifiers with their values in-place according to the bindings held in the scope used for that evaluation.

Reduction will ignore any and all transforms in the scope and must not apply them.

If all identifiers in the expression are resolved, then reduction may begin executing the operations corresponding to that expression, otherwise, if all identifiers cannot be resolved, it is an error.

Reduction of a program is performed according to the definitions of default expressions and default functions such that some implementation defined operations are executed in an implementation dependent order in a manner which produces a correct result according to those definitions provided in this document.

Contexts

The Context is used to persist resources in the environment for the execution one or more operations.

Contexts may be arbitrarily nested where the the inner context closes over the outer context.

If a new context is created, it must always close over the scope.

A new context is not required to include bindings from the scope that it closes if it can be proven that those bindings are not used when the closing context becomes the scope.

Only one context may be used as the scope for evaluationat any given time.

Executing some default expressions may change which context is the scope.

When a context is discarded, the bindings and the data forming identifiers and values are also discarded.

If a context closes a binding, then that binding cannot be discarded when the closing context is discarded.

If a context closes over a binding, and the closed context is discarded before the closing context, the binding must be perserved until the closing context is also discarded or that binding is shadowed.

Definition

Definition will create a binding in the top level.

If the Definition expression is given a symbol followed by data, it will create a binding where the symbol is the identifier and the data is the value.

If the Definition expression is given only a function of the specific type (Collection -> *), it will create a transform where the collection parameter corresponds to the type of Program Model components which are accepted as identifiers that the transform will be applied to.

Assignment

Assignment will create a binding in the scope where it is evaluated which will shadow any conflicting bindings from closed contexts.

The binding created by the Assignment will use the symbol as the identifier and the data following the keyword symbol, =, as the value.

If the symbol provided to the Assignment is already used as the identifier in a binding which is not from a closed context, it is considered an error.

Assignment will make the binding identifier available in the scope before the value on the right is evaluated such that recursive bindings may be created if the identifier is used once or more in the value.

Let

Let will create one or more bindings in a new context which will be used as the scope to evaluate the body.

Each binding will be created using the symbol preceding the keyword symbol, 'be, as the identifier and the data following it as the value.

When evaluation of the body has been completed, Let will discard the context that was used for evaluation.

Control Expressions

There are a number of default expressions which do not correspond to operations, but instead affect how a program is evaluated, in other words, they control the "flow" of evaluation. They are...

  • sequences
  • inversion
  • branching
  • quotation
  • computation

Sequences

Sequencing will transform any expression which begins with the symbol 'do and follows with some number of expressions which will be referred to as steps.

Each step will be evaluated in the order in which it occurred in the Sequence expression.

Sequencing will establish a context which will be the scope used to evaluate each step.

When evaluation of a Sequencing expression is completed, the scope which was used to evaluate its steps will be discarded.

If evaluation of a step executes an operation which changes the scope, those changes will persist until the scope is discarded.

If evaluation of a step adds a binding to the scope, that binding will shadow any bindings which the scope may have closed when it was constructed.

A step may not create a binding which shadows a binding created by a previous step. This will result in an error.

Inversion

Inversion accepts a number of symbol and function pairs which will be used to create bindings in a new context which will be used as the scope when evaluating the last sub-expression following the in symbol, referred to as the body, in the Inversion expression.

Bindings are created from the symbol and function pairs by treating each symbol as the identifier and each function immediately following that symbol as the value in a binding. Functions which are bound by Inversion in this manner are referred to as modulators.

If a modulator is called in the body, it will be applied with an additional parameter which is a function corresponding to the remaining portion of the body to be evaluated, referred to as the continuation. That function will accept one parameter which will be used as the result of the application in the body.

The scope used to evaluate the body of a continuation closes the scope used by the inversion when evaluating its body. This closure includes any changes made to the inversion scope by operations which were executed before the continuation is provided to a modulator application.

If a modulator is applied in the body of an inversion, the result of that inversion is determined according to the body of each modulator...

  1. If a modulator does not apply its continuation, the result of the inversion will be the result of the application of that modulator.
  2. If a modulator applies its continuation in its body, but not as the tail, then the continuation will be evaluated and its result used as the result of the continuation application in the modulator body, and the result of the inversion will be the result of the tail of the modulator.
  3. If a modulator applies its continuation in its body as the tail, the then the result of the inversion will be the result of the application of the continuation.

If an inversion body does not contain any modulator applications, the result of the inversion is the result of evaluating the body.

If an inversion body contains multiple modulator applications, only the last application will determine the result of the inversion.

If the inversion body applies a modulator as it's tail, then the continuation for the modulator must always result in _ when applied.

Branching

Branching accepts a series of Judgement and program pairs and an optional trailing keyword symbol, 'else, and an program.

Branching will evaluate each Judgement expression, in the order in which they occur in the Branch expression, until either a Success result occurs or there are no more Judgement expressions to evaluate.

The result of a Branch expression is determined according to the following cases...

  • If evaluating a Judgement results in a Success, then the program immediately following that Judgement is evaluated and the result is the result of the Branch expression.
  • If no evaluation results in a Success, then the trailing program is evaluated and that result is the result of the Branch expression.
  • If there is no trailing program, and no evaluation results in a Success, then the result of the Branch expression is _.

If evaluating a Judgement expression results in Success, the Branch cannot evaluate other Judgement expressions or programs.

Quoting

Quoting will capture the program given after the keyword symbol, quote, and will result in the Program Model corresponding that program.

The Program Model is formed using collections and can be used as a collection in further programs.

Computing

Computing will evaluate a program in a context.

The program supplied to the Computing expression may be an expression or a program as captured by a Quote expression.
