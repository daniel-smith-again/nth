Evaluation

1-1

Evaluation is the abstract process of replacing Program Model components with their corresponding semantic reductions.

1-2

All programs which are evaluated are comprised of expressions and data. Data will always evaluate to itself as if no evaluation had taken place. Expressions are evaluated by first rewriting and then reduction. Evaluation will always yield a data result when completed.

Rewriting

2-1

An expression which is being evaluated must first be rewritten.

2-2

An expression is rewritten by testing it against the type constraining the parameter of a transform. If the expression is an instance of the parameter constraint, then the transform is applied to the expression. If there are no transforms which can be applied to the expression, then the rewrite stage of evaluation is complete.

2-3

If a transform is applied to an expression, the expression will be replaced with result of the application.

2-4

If the result of a transform application is _, then the expression will be removed from the program and discarded.

2-5

Given an expression which is a possible parameter to more than one transform, the transform which is a subtype of every other constraint will be applied.

Reduction

3-1

Given a program which is comprised of at least one expression, that expression may be reduced according to the semantics associated with that expression.

3-2

A program consisting of only data, a symbol, number, string, or quote, will always reduce to that data. Thus no resources should be used when evaluating data.

3-3

An expression may only be reduced if it is a default expression.

3-4

An expression which is not a default expression may be rewritten if it can be accepted by a transform.

3-5

An expression which can be reduced will consist of keywords, identifiers, and other expressions.

3-6

An identifier is reduced in every case by the value bound to it in the scope.

Resolution

4-1

The semantics with which an expression is reduced may involve treating some fields as values. If those fields are not data, but are instead programs, then the process of resolution will add additional evaluations in order to reduce those fields to data before reducing the actual expression.

4-2

Given an expression which contains a field that is another expression, and that field is used as a value when rewriting the expression, the process of rewriting the expression is deferred and then field expression is evaluated first. Once evaluation of the field expression has produced a result, the outer expression will be rewritten using the result as a value in place of the field.

4-3

Given an expression which contains a field that is a symbol, and that field is used as a value when rewriting the expression, that symbol is assumed to be an identifier and the process of rewriting the expression is deferred and then the scope is searched for a binding using that identifier. When the binding is found, the expression will be rewritten using the value for that binding in place of the field.

Contexts

5-1

A context is a group of bindings which represents the values which should be accessible to evaluation when attempting to resolve an identifier.

5-2

Multiple contexts may exists during evaluation of a program.

5-3

A single context will only ever be used when resolving an identifier. At any given point during evaluation, the current context to be used is referred to as the scope.

5-4

The semantics of some default expressions may create, discard, or move contexts.

5-5

The bindings which must be included in a scope are directly correlated to the outer expressions which have created bindings before evaluating their nested expressions.

5-6

If an identifier is given in a default expression which was not previously bound in an outer expression, it is considered an error.

5-7

The semantics for some default expressions require that a separate context be created as the scope of evaluation for any resolutions and then discarded once a result is determined. In this case, the bindings in a context which is discarded may also be discarded if they are not closures.

5-8

Data which is bound in a context as a value is said to be instantiated.

Closures

6-1

A closure is a reference to a binding in one context which appears in another context as if it were a binding created in that context.

6-2

If a binding is created in a context containing a closure using the same identifier, any further resolution of that identifier shall use the value from the new binding and not the value from the closure. The value from the closure is said to be shadowed.

6-3

In the case that a default expression creates a context as the scope for resolution, that scope will include closures for all bindings which are not shadowed by resolving expressions.

6-4

In the case that a binding is discarded, but the closure referencing it is still accessible in a context, the value of the discarded binding will be copied and used as the value for the closure. The copied value will only be discarded when the context containing the closure is discarded.

6-5

In the case that multiple closures must copy a value, only one value shall ever be copied. This value is shared by every closure and cannot be discarded until every closure is discarded.

Bindings

Definition

7-1

The default expression, Definition, will create a binding in the Top Level.

Figure 2 7-2

Definition will not change the image which is the current top level.

7-3

The binding established by Definition cannot be included in any scope until the Definition expression has been evaluated.

Assignment

8-1

Assignment is a default expression which will create a new binding in the scope.

Figure 3 8-2

Assignment cannot shadow an existing binding. If the symbol given to Assignment as the identifier is already used by an existing binding in the scope, it is considered an error.

8-3

Given multiple programs which are evaluated in a single scope, an Assignment expression may be evaluated in the order in which the identifier it binds first occurs. In general, an Assignment, (x = a) and the first occurrence of the identifier, x, may be swapped such that the first occurrence of x is the Assignment expression. If the first occurrence of the identifier is not in the same scope as the Assignment, they cannot be swapped.

Let

9-1

Let is a default expression which will create a new context, establish some bindings in that context, and then use that context as the scope to evaluate an expression. The context that Let establishes as the scope is referred to as the Let scope.

Figure 5 9-2

The body of a Let expression is not required to be nested within its own expression. As presented in Figure 5, any program component following the keyword, in is treated as a component of a single expression constructed using a Deconstructing Bind.

9-3

The result of evaluating a Let expression must be the result of evaluating the body in the Let scope.

9-4

The declarations component of a Let expression is formed according to the following sequences...

  • Variable Bind: '(''Symbol = ''Program)
  • Deconstructing Bind: '(''Expression = ''Collection)
  • Symbolic Bind: '(''Symbol : ''Program)

Variable Bind

10-1

A Variable Bind sequence in a Let expression will add a binding to the Let scope such that the symbol to the left of the keyword is used as the identifier and the data to the right of the keyword is used as the value.

Deconstructing Bind

11-1

A Deconstructing Bind sequence in a Let expression will add a number of bindings to the Let scope which are fields in the collection to the right of the keyword.

11-2

The identifiers used in a Deconstructing Bind are taken from the expression given on the left of the keyword.

11-3

Each identifier is bound to one or more indicies from the destructured collection according to the following rules...

  • A symbol is bound as the identifier to the field at the index equivalent to the order in which the symbol occurred in the expression. The first symbol or _ must always correspond to the first index (1), with each additional symbol or _ corresponding to a consecutively higher index.
  • If the symbol is _, the corresponding fields will be claimed from the deconstructed collection as if the _ were another symbol, but no binding will be made. The fields which were claimed cannot be accessible as values in the Let scope. Additionally, the _ symbol may be used multiple times and each time will be interpreted as unique.
  • The keyword * signifies that the symbol will be bound to a new collection containing at least the initial field at the corresponding index and any additional fields at higher indicies which are not yet bound. The process of determining which fields are included in the new collection is not eager and if additional symbols are given in the expression after the *, then only the fields up to the k-n index may be claimed where k is the number of fields to be included in the new collection and n is the least number of fields which must be left after the highest index included by k in order for all following bindings to have values.
  • A constraint may be added, using the default expression for creating constraints, to any symbol or _. This will require that the first field which satisfies the constraint will be bound to the symbol. If the field at the corresponding index to the symbol does not satisfy the constraint, then that field will be unclaimed (and available for binding by a previous symbol) and the next consecutive index will be tested. A symbol or _ with a constraint may still be postfixed by the * keyword, but the constraint must allow a Collection or Array type otherwise, it is considered an error.
  • An expression provided inside the initial deconstructing expression requires that the field at the index corresponding to the position of the nested expression must be a collection. The expression nested inside the deconstructing expression will be treated as another deconstructing expression corresponding to the indicies in the nested collection. If a nested deconstructing expression is given, and the field at the corresponding index is not a nested collection, then the first collection at a higher index will be used, where any previous fields are unclaimed (and available for binding by a previous symbol).
11-4

A Deconstructing Bind is not required to consume every field in a collection. If a collection contains a greater number of fields than what is consumed by the Deconstructing Bind, those fields are skipped as if they were consumed by the keyword _

11-5

The collection provided to a Deconstructing Bind must minimally contain the a number of fields equivalent to each binding to be established by a Deconstructing Bind. If a collection does not have enough fields, it is considered an error.

11-6

The collection provided to a Deconstructing Bind must minimally contain fields of an amount, ordering, and type which will satisfy the constraints placed on the symbols in the deconstructing expression, otherwise it is considered an error. The binding expression in a Deconstructing Bind sequence effectively corresponds to a collection supertype which the destructured collection must be an instance to avoid error.

Symbolic Bind

12-1

A Symbolic Bind will establish an identifier in the Let scope which is constrained, but has no value. The symbol to the left of the keyword will be used as the identifier and the data to the right of the keyword will be used as the constraining type such that a Symbolic Bind sequence, x : T, establishes a constrained identifier in an identical manner to the expression, (x : T).

12-2

A Symbolic Bind must always evaluate the Let body using Abstract Evaluation

Control Expressions

Application

13-1

Application is a default expression which will apply a function to some parameters.

13-2

Application directly corresponds to the application operation.

Figure 7 13-3

Given an Application expression, the first field must resolve to a function. The following fields may resolve to anything, and must be used as parameters in the order in which they occur as fields in the expression.

13-4

The fields in an Application expression must be resolved in a consecutive order beginning with the field at the smallest index and continuing to the field with the largest index (the last field).

Sequencing

14-1

The default expression, Sequence, will evaluate multiple expressions, referred to as steps, in the order of their occurrence, in a single, newly created scope.

Figure 8 14-2

The result of evaluating a Sequence must be the result of evaluating the last expression where the last expression is one which is not followed by another , keyword.

14-3

It is considered an error if a , keyword is not followed by an expression.

14-4

As presented in Figure 8, a Sequence expression will group fields into separate expressions where each expression includes all fields up to the next , keyword. This is such that

(f x , g y , h z)

is synonymous to

((f x), (g y), (h z))

Branching

15-1

The Branching default expression will chose one of a number of programs to evaluate based on a boolean value or instance.

Figure 9 15-2

A Branch expression may only ever evaluate one Case.

15-3

The result of a Branch expression must be the result of evaluating a Case.

15-4

A Condition is either a single expression or a sequenced of a value following an expression. A Condition must always be a value followed by an expression unless the Condition value is a Boolean.

15-5

A Case is chosen by a Branch according to a supertype relation to the Condition...

  • If the Condition is a subtype of the value prefix in a Case, then that Case must be evaluated.
  • If the value of the Condition is true, then the first Case following the Condition must be evaluated.
  • If the value of the Condition is false, then the second Case following the Condition must be evaluated.
  • If the value of the Condition is false, and only one Case is given in the Branch, then the result of that Branch is _.
15-6

If no Case value is a supertype of the Condition, then the sum of every Case value must result in a supertype such that given,

((𝛜 = (type a b c d)),
 (𝛖 = (type a c)),
 (𝛓 = (type b d)))

A Condition bound to 𝚱 of the type 𝛜 such that (𝚱 : 𝛜) requires Case lists of either

(? 𝚱 a 'a b 'b c 'c d 'd) where ((a + b + c + d) = 𝛜)

or (? 𝚱 𝛖 'e 𝛓 'f) where ((𝛖 + 𝛓) = 𝛜)

Inversion

16-1

Inversion is a default expression which allows the order in which a program is evaluated to be decided by one or more arbitrary functions, thus "inverting" the normal order of evaluation.

Figure 10 16-2

An Inversion consists of two components, a Body, which is a program to be evaluated and a list of handlers, which are the functions which facilitate the inverted evaluation order.

16-3

The handler list is a sequence of symbols which must resolve to a function or a collection of named fields where each field is a function. The identifiers which bind either a function or a field which is a function are included in a new scope which is used to evaluate the body of the Inversion.

16-4

A function which is used as a handler must be a subtype of the function ((* -> *) -> *).

16-5

When a handler is applied in the Body of an Inversion, the first parameter must be a continuation corresponding to a program which is the Body where the field corresponding to the application of that handler is a parameter of the continuation given as a parameter to the handler.

16-6

The result of an Inversion expression may be one of the following...

  • The result of the Body when the Body does not contain an application of a handler.
  • The result of a handler Body when the Inversion Body applies that handler and the handler Body does not contain a containuation application as the tail.
  • The result of a continuation application when the Inversion Body applies a handler and then that handler Body applies a continuation as the tail.
